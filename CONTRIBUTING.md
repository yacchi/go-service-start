# Contributing

Thanks for taking the time to contribute!

## Reporting issues

Open a GitHub issue with:

- A clear description of the problem or proposal.
- A minimal reproducer when reporting a bug (Go version, code sample, observed vs. expected behavior).
- Relevant log output, if any.

## Development

Requires Go 1.23 or later.

```sh
go test ./...
go test -race ./...
go vet ./...
gofmt -l .
```

All checks above must pass on every supported Go version (see
[`.github/workflows/ci.yml`](.github/workflows/ci.yml)).

## Pull requests

- Keep changes focused. Unrelated cleanups belong in a separate PR.
- Update or add tests for any behavior change. The existing test suite covers
  startup ordering, shutdown ordering, error aggregation, state transitions,
  and concurrency edges — extend it rather than working around it.
- Update `README.md` and the package documentation when public API changes.
- Add an entry under "Unreleased" in [`CHANGELOG.md`](CHANGELOG.md).
- Run `gofmt -w .` before committing.

## Code generation

`servicestate_string.go` is generated by `stringer`. If you add or reorder
values in `ServiceState`, regenerate it:

```sh
go install golang.org/x/tools/cmd/stringer@latest
go generate ./...
```

## Releasing

Releases follow [Semantic Versioning](https://semver.org/). Tag commits on
`master` as `vX.Y.Z`; pkg.go.dev picks them up automatically.
